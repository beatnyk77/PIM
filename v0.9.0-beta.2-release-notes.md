# PIM v0.9.0-beta.2 Release Notes (Controlled Private Beta)

We are thrilled to announce the release of **PIM v0.9.0-beta.2**, which represents the final stabilization checkpoint and complete E2EE security hardening of our decentralized messaging system. This release elevates PIM into a production-ready, highly secure controlled private beta build.

With this release, PIM enforces absolute cryptographic boundaries, dynamic data compilation, stateless routing, and multi-tier rate limiting for secure decentralized operations without third-party tracking or cloud server dependencies.

---

## 🚀 Key Improvements & Security Advancements

### 1. Complete Removal of Mock Identities & Fallbacks
To establish mathematically secure cryptographic trust boundaries and ensure 100% Signal protocol integrity:
*   **Excised 'user2' and 'test-group' Fallbacks:** Wiped out all hardcoded mock user identifiers (`'user2'`) and static auto-join test groups (`'test-group'`) from active client layers. All group interactions are now exclusively invite-only and scannable-link driven.
*   **Fully Dynamic Inbox & SQLite Compilation:** `HomeScreen.tsx` maps active conversation rosters dynamically from database records. Conversation cards are compiled by querying all messages in WatermelonDB (`LocalDb`) and parsing Signal-derived direct channels and active MLS group sessions, sorted descending by active timestamp.
*   **Custom Start Chat Modal:** Integrated a custom, premium, cross-platform `Start New Chat` modal. Users can now enter any Registration ID/User ID dynamically, instantly spawning a secure direct messaging session.
*   **Dynamic Token Subscriptions:** Upgraded `subscribeInboundTokens()` in `MessageRelay.ts` to fetch active contacts list dynamically via `IdentityService.getContacts()` and register all contacts' inbound token batches on our backend to shield metadata from correlation.
*   **Roster Key Fallbacks:** Refactored legacy key distribution fallbacks to resolve active participants dynamically from `GroupSessionManager.getGroupRoster(groupId)` instead of utilizing static placeholders.

### 2. Hardened E2EE Cryptographic Envelopes
*   **Production PreKey Bundle Resolution:** Client screens and identity managers now query the live Server identity registries (`IdentityService` and `MessageRelay.fetchPreKeyBundle`) to retrieve active peer cryptographic prekey bundles.
*   **Base64 Signal Identity Representation:** Standardized peer-to-peer handshakes on Signal Base64 encoding parameters of local classical public identity keys (`arrayBufferToBase64(keys.identityKey)`) instead of using mock SHA-256 strings.

### 3. Backend Production Readiness & Sockets Hardening
To safeguard our stateless relay node against denial-of-service (DoS) attempts, metadata correlation attacks, and injection:
*   **Privacy-First Anonymized Request Logger:** Integrated zero-telemetry request logging middleware in `backend/server.ts` that anonymizes client IP addresses (masking IPv4 octets to `x.x`) to track server uptime and request durations without harvesting user metadata.
*   **Double-Stage Compiler Isolation (Dockerfile):** Engineered a secure, multi-stage production Docker build. Stage 1 compiles TypeScript source code inside an isolated `builder` node container. Stage 2 copies only minimal compiled production artifacts (`dist/`) and installs production dependencies (`npm ci --only=production`) inside a lightweight `node:20-alpine` runner container, keeping footprint and vulnerability surfaces minimal.
*   **tsconfig Performance Optimization:** Configured compiler options to target modern runtimes (`es2022`), specify Node module resolution explicitly, disable source maps in production, and remove comments during build.
*   **Hardened Security Headers Middleware:** Appended strict headers including `Strict-Transport-Security` with preload, CSP ancestors blocks (`frame-ancestors 'none'`), no-referrer tracking, prefetch blocks (`X-DNS-Prefetch-Control`), and hardware API locks (`Permissions-Policy`).
*   **HTTP & Socket Rate Limiting:** Enforces IP-level HTTP rate limiting (100 req/min/IP), Socket connection rate limiting (10 connections/min/IP), and Socket packet rate limiting (300 packets/min/socket) to guard against resource exhaustion.
*   **Production CORS Restrictions:** Restricts cross-origin requests securely to authorized domains parsed from `process.env.ALLOWED_ORIGINS` (defaulting to secure domains `https://app.pim-protocol.org` and `https://pim-client.netlify.app`), while allowing null origin handshakes to support native iOS and Android application containers.

### 4. Strong PIN Settings Sync
*   **Cryptographically Random PINs:** Eliminated the static transfer PIN `"123456"`. When executing a multi-device data export in settings, the app now dynamically generates a strong 8-digit random numeric passcode and stores it in transient component state, keeping imports fully secure.

### 5. Local AI Model Integrity
*   **Integrity Verification:** In `AiAdvisor`, the `verifyModelHash` function now strictly returns `false` on a SHA-256 integrity hash mismatch, immediately blocking corrupted or tampered local GGUF models from initializing.

## ⚠️ Known Limitations & Known Issues (Beta v0.9.0-beta.2)

### 🛡️ Architectural Limitations (Decentralized & Zero-Knowledge by Design)
1. **Stateless Relay Amnesia:**
   *   *Limitation:* PIM's default relay server is completely stateless and retains no database. It does not queue or store messages for offline recipients.
   *   *Consequence:* If a contact is disconnected at the moment you send a message, the packet is permanently dropped.
   *   *Mitigation:* Self-hosting a private relay with a transient PostgreSQL cache is recommended for high-availability production environments (see [relay_deployment.md](file:///Users/kartikaysharma/.gemini/antigravity-ide/brain/bff7db43-e758-455e-b241-c2f8aed24787/relay_deployment.md)).
2. **Decoy Partition Cryptographic Isolation:**
   *   *Limitation:* The Plausible Deniability Decoy Vault runs an independent SQLite instance derived from a separate PBKDF2 salt derivation chain.
   *   *Consequence:* Real database profiles, contacts, and secure keys never synchronize with the decoy database. Any data bridge would leave structural metadata footprint traces, defeating deniability.

### 🐛 Known Issues (Active Beta Resolution Path)
1. **Aggressive Background Socket Drops (iOS):**
   *   *Issue:* Due to aggressive iOS system background task termination, idle WebSockets are systematically terminated within 30–60 seconds of client backgrounding.
   *   *Status:* Under active resolution. Opening the client triggers an instant, forward-secret key-exchange handshake to restore connection. Silent APNs (Apple Push Notification service) wake support is scheduled for `v0.9.0-beta.3`.
2. **Local AI Model RAM & Thermal Spikes:**
   *   *Issue:* Loading and running the quantized local `phi-3-mini` GGUF model via the `llama.rn` engine requires at least 2.2GB of continuous resident RAM. Legacy devices may encounter battery drain, thermal throttling, or OOM crashes.
   *   *Status:* Mitigated. Users on resource-constrained hardware can toggle **Lite Mode** in settings to completely bypass local LLM memory allocations, replacing model loads with high-speed timing-shielded noise padding sweeps.

## 📋 Private Beta Launch Checklist

This checklist must be executed by deployment coordinators to validate the release readiness of the `v0.9.0-beta.2` private beta bundle:

### Phase 1: Compile & Structural Verification
- [ ] **TypeScript Static Verification:** Run `npx tsc --noEmit` in both `/app` and `/backend` to verify 0 type errors.
- [ ] **Mock Key Cleanliness Check:** Execute grep scans to ensure no occurrences of `'user2'` or mock arrays remain in active client production files (excluding test suites `FullFlowTest.ts`, `TestPhase4.ts`, and `TestPostQuantum.ts`).
- [ ] **Debug UI Review:** Confirm that no developer testing triggers or overrides (like `toggleGroupMode` or mock settings passcodes) remain in any visible client screen.

### Phase 2: Cryptographic & Security Verification
- [ ] **Dual-Layer E2EE Handshake:** Verify that a peer-to-peer session successfully establishes PreKeys using actual Base64 keys rather than local fallbacks.
- [ ] **Model Integrity Shielding:** Attempt to load a local model file with a corrupted SHA-256 hash in `AiAdvisor` and confirm that it is strictly blocked with a loading failure.
- [ ] **Dynamic PIN Sync Check:** Trigger a device settings data export, verify that a random 8-digit PIN is dynamically generated, and confirm that the previous PIN `"123456"` is completely inoperable.
- [ ] **Vault Isolation Test:** Mount the Decoy Vault using the decoy passphrase. Verify that no real threads or contacts are visible within the decoy database viewport.

### Phase 3: Sockets & Relay Deployment Verification
- [ ] **Production Relay Fallback:** Run the client with an undefined server URL env parameter and confirm it successfully falls back to connecting securely to `wss://relay.pim-protocol.net`.
- [ ] **CORS Verification:** Query the backend relay node using an unauthorized web origin and confirm the request is rejected. Confirm that native mobile null-origins are accepted.
- [ ] **IP-Masked Logger Check:** Make requests to the backend server and verify that standard output logs list anonymized client IPs (e.g. `192.168.x.x` or `ipv6-masked`) instead of full raw addresses.
- [ ] **HTTP Rate Limiter Test:** Fire 120 rapid API requests from a single IP to `/api/prekey` and verify that the backend responds with HTTP 429 (Too Many Requests) after request 100.
- [ ] **Socket Packet Flood Test:** Attempt to send 350 socket packets in under 60 seconds from a single connection and verify that the socket is instantly disconnected by the server's rate limiter.

### Phase 4: Signature & Build Verification
- [ ] **EAS Build Configuration:** Verify `app/eas.json` production build profiles are locked to strict target devices.
- [ ] **Cryptographic Signing (Android):** Verify that the Android release APK is compiled with a valid production keystore and signed using `apksigner` (or Expo EAS credentials).
- [ ] **Cryptographic Signing (iOS):** Confirm that the iOS release build (`.ipa`) is compiled using our private enterprise provisioning profile and signed with a valid developer certificate.
